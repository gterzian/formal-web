- For concurrent spec algorithms such as fetch-and-wait navigation steps, prefer modeling the pause point as explicit pending state on `UserAgent` plus a separate resume transition before introducing real runtime tasks or I/O.
- When a spec algorithm pauses and later resumes after a wait point, model the resumed portion as an explicit continuation helper instead of re-entering the top-level algorithm at a later argument state.
- When a spec wait has multiple wakeup conditions, model each wakeup reason explicitly in the LTS; if one branch produces no result and just returns, represent that as its own continuation path instead of folding it into the response-arrival case.
- Model `beforeunload` as its own user-agent and event-loop command path. Queue a dedicated runtime task instead of encoding it as a synthetic UI event, and let the completion handler resume navigation and log any resulting navigation failure.
- Split cross-document navigation after fetch response into two phases: populate/load should install a pending finalization record, and only the content runtime's explicit `FinalizeNavigation` signal should commit session history, switch the active document, and queue the first render for the new page.
- When routing a navigation response into document loading, match HTML on the MIME type essence rather than the exact header string so parameters such as `charset` still take the HTML loading path.
- After a navigation response installs a new `Document`, derive follow-up event-loop and content-process dispatches from the updated `UserAgent` state so the dispatched `DocumentId` matches the newly active document.
- Introduce a small LTS action type above navigation helpers so spec-visible concurrent steps such as "begin navigation" and "fetch response arrives" are explicit labels, while helper functions remain implementation detail under those labels.
- It is acceptable for the LTS layer to factor a convenience spec helper into multiple explicit labels, such as separating top-level traversable creation from the later begin-navigation and fetch-completion steps.
- If a spec convenience helper only bundles multiple LTS-visible steps, prefer modeling those steps directly in the action system and omit the convenience helper unless it still carries independent explanatory value.
- If a spec algorithm is only a top-level entry point and is not referenced by other spec algorithms, it does not need to be preserved as a separate helper in the model when the LTS already captures its intended behavior.
- For WriterT/state refinement proofs, keep runtime helpers definitionally transparent and prove small projection lemmas for packed returns such as emitted-effect arrays or next-state fields instead of fighting large `simp` goals over the whole monadic term.
- Queue update-the-rendering tasks in `UserAgent` without deduplicating them there, and let `EventLoop` own the coalescing policy for both `DispatchEvent` bursts and `UpdateTheRendering` work; at each turn, collapse every queued `DispatchEvent` into one retained-per-type batch at the position of the first queued dispatch task, then collapse each contiguous `UpdateTheRendering` run to one task before taking the head of the normalized queue. The embedder forwards raw UI events and rendering-opportunity notifications without local batching, while content command completion only wakes the event-loop worker.
- Route content-process notifications to the component that owns the resulting state transition instead of forcing them through `EventLoop`; navigation requests, `beforeunload` completion, and finalize-navigation can go straight to `UserAgent`, while document-fetch requests stay on the owning `EventLoop` so fetch start/completion routing and future cancellation state remain local to that worker. Direct content-to-embedder traffic stays limited to painted frames.
- Keep `UserAgent` writer sites explicit: construct `EventLoopTaskMessage` values at the `tell #[.eventLoopMessage ...]` call site instead of hiding them behind option-returning dispatch helpers, and reserve `Runtime` in helper names for code that still handles a distinct runtime-only message type.
- In Lean control flow that exits early on missing state, prefer `let some value := option | ...` over nested `match` blocks when the `none` branch just logs, restores state, or returns; use that style especially around `UserAgent` task handlers so the main path stays visible.
- Keep `FormalWeb.Runtime` limited to starting the top-level user-agent, fetch, and timer workers plus holding their top-level channels; let `UserAgent` own event-loop worker lifecycle, and let each `EventLoop` worker talk directly to fetch/timer workers through the channels cloned into it.
- Model HTML timer waiting in `FormalWeb.Timer`, use `EventLoop` runtime effects to schedule and clear `run steps after a timeout`, and route timer completions back into the owning event loop as explicit task messages.
- Keep cross-document session-history commit/finalization helpers in `FormalWeb.SessionHistoryNavigation`; model them against navigable/session-history state there instead of embedding that logic in runtime-facing `UserAgent` code.
- Route Rust-originated navigation starts through an event-loop message keyed by source navigable identity, run the pre-in-parallel start-navigation steps immediately when that message is handled instead of queueing a synthetic event-loop task, and let the originating event loop resume after the `UserAgent` reaches the in-parallel handoff; likewise, queue fetch-response document loading back onto the active window's event loop as a Lean task before emitting content commands.
- Do not let unknown empty-target source navigables fall back to top-level traversable creation; child iframe navigables need an explicit model instead of silently reusing top-level navigation selection.
- Keep `_iframe|...` helper traversables out of `UserAgent`'s browser-ui active selection; embedder `DispatchEvent` and `renderingOpportunity` messages intentionally route through the visible top-level traversable.
- Treat empty-target `browser-ui` navigations as top-level replacement: drop the old top-level traversable subtree and its pending navigation state before bootstrapping the new top-level traversable.
- Retire `_iframe|parent|source` helper traversables from explicit content-side iframe removal or parent-document-destroy notifications, not from parent `FinalizeNavigation`; replacement pages can create new helpers before the old parent navigation commits.
